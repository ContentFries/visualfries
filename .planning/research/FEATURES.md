# Feature Landscape: Deterministic Frame Provider API

**Domain:** Server-side deterministic frame extraction from an animation library  
**Researched:** 2026-02-21  
**Confidence:** HIGH (codebase verified; Remotion/Motion Canvas sourced from Context7 + official docs; ecosystem patterns cross-referenced)

---

## How Comparable Tools Structure This Problem

Before the feature categories, here is what was learned from studying Remotion and Motion Canvas — the two closest comparators.

### Remotion (`@remotion/renderer`)

Remotion exposes `renderFrames()` with an `onFrameBuffer` callback (when `outputDir: null`) that receives a raw `Buffer` for each frame.
Key observations:

- **No provider abstraction.** The caller bundles code, Remotion orchestrates headless Chrome, and frames come back to the caller via callback. There is no injectable "give me frame X for component Y" interface. The provider _is_ Remotion itself.
- **Frame-by-frame callback is the ergonomic primitive.** `onFrameBuffer` (v3.0+) is the mechanism consumers rely on for in-memory pipelines.
- **Per-frame timing data is a first-class return.** `slowestFrames` (array of `{frame, time}`) ships in the return value. `onFrameUpdate(framesRendered, frame, timeToRenderInMilliseconds)` fires per frame.
- **Media cache is caller-controlled.** `mediaCacheSizeInBytes` and `offthreadVideoCacheSizeInBytes` are explicit knobs. Remotion does not deduplicate frames automatically; the caller owns that.
- **`everyNthFrame`** is the "skip duplicate frames" mechanism — renders every Nth frame, designed for GIF output. This is crude and static; it does not detect actual visual identity.
- **Strict vs graceful degradation is implicit.** No strict/diagnostics mode. Errors surface as thrown exceptions or `RendererResult.Error`.
- **`frameRange?: [number, number]`** was the last parameter added (v4.0.421). Range-bounded rendering was an afterthought; it is not the primary loop primitive.

### Motion Canvas (`Renderer` + `Exporter` interface)

Motion Canvas uses a push-model: `Renderer.render()` drives the scene through a playback loop, calling `exporter.handleFrame(canvas, frame, sceneFrame, sceneName, signal)` per frame. Source: verified from `Renderer.ts` and `Exporter.ts` raw source.
Key observations:

- **The Exporter interface is the provider pattern.** `ExporterClass.create()` + `handleFrame()` is structurally equivalent to a frame provider. But the dependency points the other way — the renderer pushes frames _to_ the exporter rather than the exporter _pulling_ frames.
- **`canvas` (HTMLCanvasElement)** is the frame payload. The exporter converts it to whatever format it needs (`canvas.toDataURL(mimeType, quality)` in `ImageExporter`). No blob-first optimization; toDataURL is base64 by default.
- **`sceneFrame` vs `frame`** — Motion Canvas separates the absolute frame number from the within-scene frame number. This is useful when the animation has multiple scenes with independent timelines.
- **Back-pressure is manual.** `ImageExporter` has a `EXPORT_FRAME_LIMIT = 256` sentinel and polls with `setTimeout` to avoid flooding the server. No built-in async signaling.
- **No diagnostics, no cacheKey concept.** Cache-key deduplication does not exist in Motion Canvas; frames are pushed regardless of visual change.
- **AbortSignal is threaded through.** `handleFrame(canvas, frame, sceneFrame, sceneName, signal)` — the abort signal arrives at the frame handler, enabling clean cancellation mid-stream.
- **`Renderer.renderFrame(settings, time)`** is the single-frame path for stills. This is separate from the bulk render path.

### What Both Lack (= Our Differentiator Space)

- Neither has a **pull-based provider** where the library asks "give me the pixel data for component X at frame N" and the caller responds. Both assume the renderer _is_ the pixel source.
- Neither has **cacheKey-based deduplication** at the provider level.
- Neither has **diagnostics mode** (hit/miss tracking, cache ratio, per-frame latency).
- Neither separates **component-level** frame overrides from scene-level renders.

## Differentiators

_Features that set this API apart from Remotion / Motion Canvas naive approaches. Not expected by users who haven't seen them, but immediately valued once discovered._

| Feature                                                                           | Value Proposition                                                                                                                                                                                                                                                                                                                                       | Complexity | Dependencies                                                      | Notes                                                                                                                                                                                                                                                                                                |
| --------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- | ----------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **cacheKey-based frame deduplication**                                            | When two consecutive frames of a video component are visually identical (same source frame), the provider returns the same cacheKey. `RenderFrameCommand` detects this and returns the cached blob without re-rendering. This is the "skip duplicate frames" feature done correctly — not Remotion's crude `everyNthFrame`.                             | Medium     | DeterministicFramePayload, RenderFrameCommand                     | cacheKey should be `string`. The guard in `RenderFrameCommand` already checks `isDirty`; it must be extended to also check if _all active components' cacheKeys are unchanged_ since last render.                                                                                                    |
| **Component-level resource cache** (`pixiResource` dedup)                         | DeterministicMediaManager holds a per-component resource cache keyed by `cacheKey`. If the same cacheKey re-appears (e.g., a clip loops), the PixiJS TextureSource is reused without recreating it. Native Remotion/Motion Canvas have no equivalent — they re-decode on every frame.                                                                   | Medium     | DeterministicMediaManager                                         | The resource cache must be invalidated on `destroy()` and on explicit `cacheKey` change. Memory bound: cap at reasonable max entries (not caller-configurable in v1).                                                                                                                                |
| **Diagnostics mode** (`hit/miss`, `cacheRatio`, `latency`)                        | Allows cf-worker to detect provider bottlenecks and cache thrash without adding external instrumentation. Disabled by default; zero overhead when off; never throws when on.                                                                                                                                                                            | Medium     | DeterministicMediaConfig, DeterministicMediaManager               | Comparable tools have `slowestFrames` (Remotion) and `TimeEstimator` (Motion Canvas). Neither exposes cache hit ratio. Design: diagnostics accumulate in a `DiagnosticsReport` object exposed via `SceneBuilder.getDiagnosticsReport()`. Suppress silently on any internal error.                    |
| **Strict mode** (throws on missing override)                                      | During development, if a VIDEO/GIF component has no registered override and `strict: true`, throw a descriptive error immediately. This makes "I forgot to wire a component" a hard failure rather than a silent native fallback that looks correct until it renders incorrectly on frame 1.                                                            | Low        | DeterministicMediaConfig                                          | Error message must name the `componentId` and `componentType`. Production usage should leave `strict: false`.                                                                                                                                                                                        |
| **`releaseComponent(componentId)`** on the provider interface                     | Lets the provider eagerly free resources for a component that has left the visible timeline, without waiting for `destroy()`. Remotion and Motion Canvas have no equivalent at the component level.                                                                                                                                                     | Low        | DeterministicFrameProvider                                        | If the provider does not need it, the implementation is `() => {}`. But the interface must declare it so callers who do need it have a typed hook.                                                                                                                                                   |
| **Multiple payload variants** (`url` \| `blob` \| `arraybuffer` \| `imageBitmap`) | Different consumers have different optimal transports. cf-worker prefers blob. A CDN-backed provider would return a URL. A GPU pipeline might prefer ImageBitmap. The library normalizes all of them into a PixiJS `TextureSource`.                                                                                                                     | Medium     | DeterministicMediaManager                                         | URL variant must trigger `Texture.fromURL()` (existing Pixi pattern). ArrayBuffer and ImageBitmap variants need conversion to `TextureSource` via `Texture.from()`. Blob needs `createObjectURL` → same path as URL. All paths must revoke object URLs after texture creation to avoid memory leaks. |
| **Effects remain native** (split-screen, blur work with overridden textures)      | If a VIDEO component uses `layoutSplit` or `fillBackgroundBlur`, those effects must work correctly even when the texture source is a provider blob rather than a live video element. The deterministic hook writes `pixiResource` before `PixiTextureHook` / `PixiSplitScreenDisplayObjectHook` run, so they receive the correct texture transparently. | Medium     | ComponentDirector hook priority, PixiSplitScreenDisplayObjectHook | This is only possible because of hook priority ordering. Document this dependency explicitly — it is easy to break if someone reorders hooks.                                                                                                                                                        |

## Anti-Features

_Things that look like good ideas but create problems. Deliberately exclude from v1._

| Anti-Feature                                          | Why Avoid                                                                                                                                                                                                                                                                                                                                              | What to Do Instead                                                                                                                                                                                                                                |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----- | ------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **Parallel frame fetching inside `renderFrameRange`** | It sounds like a performance win to fetch frame N+1 while rendering frame N. But Pixi renders are sequential (single-threaded RAF), and parallel provider calls with out-of-order blob arrival introduce non-determinism. Remotion splits this into `framesPerLambda` chunks across _separate processes_; it does not parallelize within one renderer. | Keep `renderFrameRange` strictly sequential: seek → request → write pixiResource → render → emit. External parallelism via multiple `SceneBuilder` instances is out of scope and is cf-worker's responsibility.                                   |
| **Provider retry / timeout logic inside the library** | Retry logic depends on transport semantics (WebSocket, HTTP, IPC) that the library must not know about. Remotion's `maxRetries` is on the Lambda invocation, not inside the renderer. Motion Canvas threads an `AbortSignal` to handle cancellation, not retries.                                                                                      | Surface provider errors as thrown exceptions. Let cf-worker own retry and timeout. The library may expose `cancelSignal?: AbortSignal` on `renderFrameRange` to support clean cancellation without retrying.                                      |
| **Built-in disk / S3 output path**                    | Remotion's `outputDir` and Lambda's `outName → S3` are heavily used but inappropriate for a library — they bake in deployment choices. Motion Canvas's `ImageExporter` uses HMR sideband (`import.meta.hot`) to write frames to disk.                                                                                                                  | The `onFrame` callback is the only output mechanism. Callers write to disk, pipe to WebSocket, or buffer in memory as they see fit.                                                                                                               |
| **`everyNthFrame`-style static skip**                 | Remotion's `everyNthFrame` is designed for GIF frame reduction. Using static skip for video deduplication produces artifacts on variable-content clips.                                                                                                                                                                                                | Use cacheKey comparison instead. If two adjacent frames share a cacheKey, the cache hit returns the previous blob. No static skip needed.                                                                                                         |
| **Global/singleton provider registry**                | Some SSR frameworks (Next.js pages, Remix loaders) use module-level singletons that leak across requests. A global `DeterministicFrameProvider` registry would share state across concurrent renders, causing frame contamination.                                                                                                                     | Provider is scoped per `SceneBuilder` instance via DI container. `setDeterministicFrameProvider()` writes into `DeterministicMediaManager` which is a per-container singleton. Two concurrent `SceneBuilder` instances get independent providers. |
| **Auto-enable when provider is set**                  | It is tempting to set `enabled: true` automatically when a provider is registered. But this would silently activate server behavior for any consumer who passes a provider to a client-mode scene.                                                                                                                                                     | Keep `enabled` an explicit config field. Registering a provider without `enabled: true` is a no-op. The validation should log a warning in development (not throw).                                                                               |
| **`contentType` auto-detection from payload**         | Remotion added `contentType` to `renderStill()` return value in v4.0.426. Detecting content type from blob MIME type at render time requires inspecting the blob header.                                                                                                                                                                               | Payload type is declared by the variant discriminant (`'blob'                                                                                                                                                                                     | 'url' | 'arraybuffer' | 'imageBitmap'`). Pixi handles codec detection internally. Do not add a `contentType`field to`DeterministicFramePayload` in v1. |
| **Audio frame extraction**                            | Motion Canvas's `Exporter.start(sounds, duration)` handles audio separately from video frames. Remotion separates `assetsInfo` (audio mixing data) from frame buffers. Trying to include audio timing in the deterministic frame API would require synchronizing two async streams.                                                                    | Audio is out of scope for v1. The provider interface covers VIDEO and GIF only. AUDIO components continue on the native `MediaHook` path regardless of `deterministicMedia.enabled`.                                                              |

## Potential Gaps (Things the Spec Might Regret Not Including)

_Features absent from the current spec that comparable tools needed later, or that the described usage pattern implies will be needed._

| Gap                                                       | Why It Matters                                                                                                                                                                                                                                                                                                                                                                                                                       | Risk if Absent                                                                                                                          | Recommended Approach                                                                                                                                                                                                                                                                   |
| --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`AbortSignal` / `cancelSignal` on `renderFrameRange`**  | Motion Canvas threads an `AbortSignal` through every `handleFrame` call. Remotion has `makeCancelSignal()`. A long render loop (e.g., 500-frame scene) with no cancellation path means cf-worker must destroy the entire `SceneBuilder` instance to stop a runaway render — a blunt instrument that also loses the DI container.                                                                                                     | HIGH — cf-worker needs to be able to stop a render mid-stream without tearing down the scene.                                           | Add `cancelSignal?: AbortSignal` to `renderFrameRange` options. Check `signal.aborted` before each frame. On abort, resolve with partial results (frames emitted so far) and set `aborted: true` in the return summary.                                                                |
| **Render summary / stats object from `renderFrameRange`** | Remotion returns `{ buffer, slowestFrames }`. Motion Canvas returns `RendererResult` enum + logs. The spec describes `onFrame` callback only, with no return value from `renderFrameRange` itself. In production, cf-worker needs to know: how many frames were emitted, were any frames skipped (cache hits), was the render aborted?                                                                                               | MEDIUM — without this, cf-worker must maintain its own counter in the `onFrame` closure and has no clean way to detect partial renders. | Return `Promise<RenderFrameRangeSummary>` where the summary is `{ framesRendered: number, framesSkipped: number, aborted: boolean, diagnostics?: DiagnosticsReport }`.                                                                                                                 |
| **`onFrame` async back-pressure**                         | Motion Canvas's `ImageExporter` uses an explicit `EXPORT_FRAME_LIMIT` sentinel + polling to prevent flooding the export server. If `onFrame` is fire-and-forget (non-awaited), cf-worker's WebSocket send queue could grow unboundedly on fast renders.                                                                                                                                                                              | MEDIUM — in practice cf-worker renders one tab at a time, but the pattern should be correct.                                            | Make `renderFrameRange` await the `onFrame` callback before advancing to the next frame. The return type of `onFrame` should be `void                                                                                                                                                  | Promise<void>`. Sequential await is sufficient; no explicit back-pressure counter needed.                                                                                     |
| **`DeterministicMediaConfig.provider` field**             | The spec lists `provider` as a field of `DeterministicMediaConfig`. But the spec also has `SceneBuilder.setDeterministicFrameProvider()`. If `provider` can be passed via config _and_ set via the setter, there must be a clear precedence rule.                                                                                                                                                                                    | LOW — but undefined behavior if both are used.                                                                                          | Config-time `provider` is the initial value. Setter overrides. Document this explicitly. In `DeterministicMediaManager`, setter always wins.                                                                                                                                           |
| **`frameIndex` zero-indexing vs one-indexing**            | Motion Canvas uses zero-indexed absolute frames. Remotion uses zero-indexed frames. `renderFrameRange(fromFrame: 0, toFrame: 30)` — is frame 0 included, frame 30 included, or exclusive? The spec says "frame count = `toFrame - fromFrame`", which implies `[fromFrame, toFrame)` (exclusive end), matching Remotion's convention. But this must be documented explicitly.                                                         | MEDIUM — off-by-one here breaks every downstream muxer.                                                                                 | Document `[fromFrame, toFrame)` explicitly in the interface and enforce it with a test: `renderFrameRange(0, 30)` must emit exactly 30 frames (0..29).                                                                                                                                 |
| **Resource leak from object URLs**                        | When the provider returns a blob, `DeterministicMediaManager` must call `URL.createObjectURL(blob)` to get a URL for Pixi's `Texture.fromURL()`. Object URLs must be revoked after the texture is created. If revocation is forgotten, long render sessions (hundreds of frames) accumulate object URLs and exhaust browser memory. Neither Remotion nor Motion Canvas faces this because they don't use blob-to-texture conversion. | HIGH — this is a memory leak vector unique to this design.                                                                              | Revoke the object URL in `handleFrame` immediately after `await Texture.fromURL(objectUrl)` resolves. Add a test: render 100 frames with blob payloads, assert no stale object URLs remain (check `URL.revokeObjectURL` call count).                                                   |
| **`ReplaceSourceOnTimeCommand` fate decision**            | The spec defers this to planning. But the command exists at `src/lib/commands/ReplaceSourceOnTimeCommand.ts` as an empty stub with a base64-only signature. If it is wired as a "compatibility wrapper" into `DeterministicMediaManager`, its current base64-only input conflicts with the new blob-first payload model. Leaving it as a stub while adding the new API creates a confusing public surface.                           | MEDIUM — if cf-worker has any existing calls to `replaceSourceOnTime()` they will silently no-op.                                       | During planning: either (a) implement it as a thin `DeterministicMediaManager` wrapper that accepts base64, converts to blob internally, and calls through; or (b) deprecate it entirely and document migration to `setDeterministicFrameProvider`. Do not ship both as co-equal APIs. |
| **Diagnostics access surface**                            | The spec says "diagnostics mode: provider hit/miss, cache ratio, latency — disabled by default, never throws." But the spec does not say _how_ the consumer reads the diagnostics. Is it an event? A getter? Appended to the render summary?                                                                                                                                                                                         | LOW — not blocking, but will be the first question asked post-ship.                                                                     | Expose as `SceneBuilder.getDiagnosticsReport(): DiagnosticsReport                                                                                                                                                                                                                      | null`. Returns `null`when`diagnostics: false`. The report is a snapshot; calling it multiple times returns the accumulated totals since provider registration (not per-call). |

## Feature Dependencies

```
DeterministicMediaConfig
  └── deterministicMedia.enabled
  └── deterministicMedia.strict
  └── deterministicMedia.diagnostics
  └── deterministicMedia.provider (optional, overridden by setter)

DeterministicFrameProvider (interface)
  └── getFrame(DeterministicFrameRequest) → Promise<DeterministicFramePayload>
  └── releaseComponent(componentId)
  └── destroy()

DeterministicFrameRequest
  └── componentId, componentType, frameIndex, fps, width, height

DeterministicFramePayload
  └── { type: 'blob', blob: Blob, cacheKey: string }
      (or url / arraybuffer / imageBitmap variants)

DeterministicMediaManager  [singleton, per DI container]
  ├── holds provider ref (from config or setter)
  ├── holds per-component resource cache (Map<cacheKey, TextureSource>)
  ├── strict flag → throws DeterministicStrictError if getFrame returns null
  ├── diagnostics flag → accumulates DiagnosticsReport (never throws)
  └── depends on: StateManager (environment guard), MediaManager (release pattern)

DeterministicMediaFrameHook  [transient, one per component]
  ├── priority < PixiTextureHook, PixiVideoTextureHook (must run first)
  ├── types: ['setup', 'update']  (no 'destroy' — manager owns cache)
  ├── calls DeterministicMediaManager.getFrameForComponent()
  ├── writes result into context.setResource('pixiResource', ...)
  └── server-only: short-circuits in client environment

ComponentDirector.constructVideo()  [routes to deterministic hook in server mode]
  └── depends on: DeterministicMediaFrameHook availability in DI

SceneBuilder.setDeterministicFrameProvider()
  └── calls DeterministicMediaManager.setProvider()

SceneBuilder.renderFrameRange(fromFrame, toFrame, options, onFrame)
  ├── seek(frameTime) → marks dirty → DeterministicMediaFrameHook fires per component
  ├── renderFrame(format, quality) → returns blob
  ├── emits onFrame(frameIndex, blob)
  ├── advances frame
  └── returns RenderFrameRangeSummary { framesRendered, framesSkipped, aborted, diagnostics? }

RenderFrameCommand  [cache guard extended]
  └── must include deterministic fingerprint: all active components' cacheKeys
      → if cacheKeys unchanged AND isDirty=false → return cached frame
```

## MVP Recommendation

### Must Have (v1 Ships With)

1. `DeterministicFrameProvider` interface + `DeterministicFrameRequest` + `DeterministicFramePayload` (all variants)
2. `DeterministicMediaConfig` type + `createSceneBuilder` config extension
3. `DeterministicMediaManager` (enabled gate, provider wiring, per-component resource cache, strict mode, diagnostics accumulator)
4. `DeterministicMediaFrameHook` (server-only, priority before texture hooks)
5. `ComponentDirector` route update for VIDEO + GIF in server mode
6. `SceneBuilder.setDeterministicFrameProvider()` + `getDeterministicFrameProvider()`
7. `SceneBuilder.renderFrameRange()` with sequential `await onFrame`, abort signal, and summary return
8. Extended `RenderFrameCommand` cache guard (cacheKey fingerprint)
9. Object URL lifecycle management (create → texture → revoke)
10. Unit tests: cacheKey change → recreation; no-op on same cacheKey; null → native fallback; strict throws; diagnostics accumulates
11. Integration tests: frame count = `toFrame - fromFrame`; blob ≥ 30% faster than base64; effects (split-screen, blur) correct with overridden texture

### Defer to Post-v1

- **Audio frame extraction** — separate problem domain
- **In-library provider retry/timeout** — cf-worker's responsibility
- **Built-in disk/S3 output** — anti-feature
- **Parallel frame fetching inside the library** — anti-feature
- **`contentType` auto-detection** — not needed until CDN integration
